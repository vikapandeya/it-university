🌐 Networking Questions

What is the difference between TCP and UDP? Easy▼

TCP (Transmission Control Protocol): Connection-oriented. 3-way handshake (SYN→SYN-ACK→ACK). Guaranteed delivery, ordering, error correction. Used for HTTP, SSH, FTP, databases. Slower due to overhead.

UDP (User Datagram Protocol): Connectionless. No handshake. No guaranteed delivery. Faster. Used for DNS, streaming, VoIP, gaming, QUIC. Application handles reliability if needed.

Explain the OSI model layers and give a real example for each. Medium▼
  1. Physical — Cables, fiber, radio waves
  2. Data Link — Ethernet frames, MAC addresses, switches
  3. Network — IP routing, routers
  4. Transport — TCP segments, UDP datagrams, ports
  5. Session — SSL session management, NetBIOS
  6. Presentation — TLS encryption, data encoding, compression
  7. Application — HTTP, DNS, FTP, SMTP
What happens when you type google.com in a browser? Medium▼
  1. Browser checks cache for DNS record
  2. OS checks /etc/hosts file
  3. DNS resolver query: stub → recursive → root → TLD (.com) → authoritative nameserver
  4. IP returned (e.g., 142.250.80.46)
  5. TCP connection: SYN → SYN-ACK → ACK (3-way handshake)
  6. TLS handshake: certificates exchanged, session key derived
  7. HTTP GET / request sent
  8. Server responds with HTML; browser parses, downloads resources, renders page
What is subnetting? Calculate the network for 192.168.10.50/26. Medium▼

Subnetting divides a network into smaller sub-networks. /26 = 26 network bits, 6 host bits = 64 addresses per subnet (62 usable).

192.168.10.50/26: Subnet boundaries every 64: 0, 64, 128, 192. 50 falls in the 0-63 range.

  • Network: 192.168.10.0
  • Broadcast: 192.168.10.63
  • Usable hosts: 192.168.10.1 – 192.168.10.62
  • Subnet mask: 255.255.255.192
What is the difference between a hub, switch, and router? Easy▼
  • Hub (Layer 1): Broadcasts all traffic to all ports. No intelligence. Legacy device.
  • Switch (Layer 2): Forwards frames based on MAC address table. Only sends to the correct port. VLANs supported.
  • Router (Layer 3): Routes packets between networks using IP addresses. NAT, DHCP, inter-VLAN routing.
How does ARP work and what is ARP poisoning? Medium▼

ARP (Address Resolution Protocol): Maps IP → MAC. Device broadcasts "Who has 192.168.1.1?" — that IP's owner replies with its MAC. Cached in ARP table.

ARP Poisoning (Spoofing): Attacker sends fake ARP replies, mapping their MAC to another device's IP. This poisons the victim's ARP cache, redirecting traffic through the attacker (MITM). Prevention: Dynamic ARP Inspection (DAI) on managed switches, static ARP entries.

What is BGP and why is it called the routing protocol of the internet? Hard▼

BGP (Border Gateway Protocol) is an EGP (Exterior Gateway Protocol) that manages how packets are routed between Autonomous Systems (ASes) — large networks run by ISPs, cloud providers, enterprises.

Called "the routing protocol of the internet" because: it maintains the internet's global routing table (~900K prefixes), ISPs exchange routes via BGP peering/transit, and route selection uses path vector algorithm (AS_PATH, MED, local preference attributes).

BGP hijacking: attacker announces more specific prefixes to attract traffic (BGP has no authentication by default). Fix: RPKI (Route Origin Validation).

Explain TCP flow control and congestion control. Hard▼

Flow Control: Prevents sender from overwhelming the receiver. Receiver advertises a window size (rwnd) — max bytes it can buffer. Sender cannot send more than rwnd unacknowledged bytes.

Congestion Control: Prevents sender from overwhelming the network. TCP algorithms:
- Slow Start: start with cwnd=1 MSS, double per RTT until ssthresh
- Congestion Avoidance: increase cwnd linearly (+1 per RTT) above ssthresh
- Fast Retransmit/Recovery: on 3 duplicate ACKs, halve cwnd, retransmit lost segment

🐧 Linux Questions

What is a zombie process and how do you handle it? Medium▼

A zombie process has finished execution but still has an entry in the process table because its parent hasn't read its exit status via wait(). Shows as 'Z' in ps output.

ps aux | awk '$8 == "D" || $8 == "Z" {print}'

Fix: Kill the parent process (init/systemd will reap orphans automatically). If parent is misbehaving: kill -SIGCHLD parent_pid to prompt it to reap children.

How does Linux boot process work from power-on to login prompt? Medium▼
  1. BIOS/UEFI: POST, finds bootable device
  2. GRUB2: Loads kernel (vmlinuz) + initramfs into RAM
  3. Kernel: Initializes hardware, mounts initramfs as temporary /
  4. initramfs: Loads drivers, finds real root filesystem
  5. systemd (PID 1): Reads unit files, starts services in dependency order
  6. Login: getty on TTY, or display manager for GUI
Explain inode exhaustion — how do you diagnose and fix it? Hard▼

Each file needs an inode. If all inodes are used, you get "No space left on device" even when disk has free space.

df -i    # check inode usage (100% = problem)
find / -xdev -printf '%h\n' | sort | uniq -c | sort -rn | head -20  # find dirs with most files

Common cause: Thousands of small cache/session/temp files in one directory. Fix: delete old files, increase inode count (requires reformatting), or move files to a filesystem with more inodes.

What is the difference between hard links and symbolic links? Easy▼
  • Hard link: Points directly to the inode. Same inode number as original. Both are equal — deleting "original" doesn't break the link. Cannot span filesystems. Cannot link directories.
  • Symbolic (soft) link: Points to the file path. Different inode. Dangling if original deleted. Can span filesystems. Can link directories.
ln /etc/nginx/nginx.conf /backup/nginx.conf      # hard link
ln -s /etc/nginx/nginx.conf ~/nginx-link.conf   # symlink
How does SELinux work and when should you use permissive mode? Hard▼

SELinux uses Mandatory Access Control (MAC). Every process and file has a security context (user:role:type:level). Policy rules define what types can access what other types (Type Enforcement).

Enforcing: Blocks AND logs violations. Production mode.

Permissive: Logs but does NOT block. Use temporarily to diagnose what would be denied: setenforce 0. Examine: ausearch -m avc -ts recent. Fix denials with audit2allow. Return to enforcing: setenforce 1.

Never leave permissive in production — set in /etc/selinux/config for permanence.

A server has high load average but low CPU usage. What do you investigate? Hard▼

Load average counts ALL runnable + uninterruptible sleep (D-state) processes. High load + low CPU = processes blocked on I/O (disk or network).

uptime           # load average
iostat -x 1      # check %util, await — is disk saturated?
iotop -ao        # which processes are doing I/O?
ps aux | awk '$8 == "D" {print}'  # D-state processes
vmstat 1         # check wa (I/O wait) column

Fix: find the I/O bottleneck (slow disk, NFS timeout, DB lock). Consider faster storage, I/O scheduler tuning, or caching.

⚙️ DevOps Questions

What is the difference between blue/green and canary deployments? Medium▼

Blue/Green: Run two identical environments. Blue = current live, Green = new version. Switch all traffic at once (DNS or load balancer cutover). Easy rollback (switch back). Requires 2x infrastructure.

Canary: Route small percentage (e.g., 5%) of traffic to new version. Monitor metrics. Gradually increase if healthy. Affects only a subset of users if issues arise. Requires traffic splitting (weighted routing, feature flags, or service mesh).

What is GitOps and how does ArgoCD implement it? Medium▼

GitOps principles: 1) Git is the single source of truth for declarative infrastructure. 2) Changes made through PRs/commits (no direct kubectl). 3) Approved changes automatically applied. 4) Continuous reconciliation detects and fixes drift.

ArgoCD implementation: Watches a Git repo. Compares desired state (Git) with actual state (Kubernetes). Syncs automatically (or on demand). Shows drift. Supports self-heal (auto-reverts manual changes) and prune (deletes resources removed from Git).

Explain the DORA metrics and what each measures. Easy▼
  • Deployment Frequency: How often you deploy to production. Elite = multiple times/day.
  • Lead Time for Changes: Time from commit to production. Elite = <1 hour.
  • Change Failure Rate: % of deployments causing incidents. Elite = 0-15%.
  • Time to Restore Service: How quickly you recover from incidents. Elite = <1 hour.
How would you implement secret management in a CI/CD pipeline? Hard▼
  • Never store secrets in Git (even in .gitignore — use git-secrets or truffleHog to scan)
  • Use GitHub Actions/GitLab secrets or Vault for CI secrets injection
  • For AWS: OIDC identity federation (no static access keys) + Secrets Manager
  • In runtime: Kubernetes ExternalSecrets Operator + AWS Secrets Manager or HashiCorp Vault
  • Rotate secrets regularly, audit access, alert on direct secret reads
  • Separate secrets by environment (dev/staging/prod have different credentials)
What is the difference between Docker and a VM? Easy▼
  • VM: Full OS virtualization via hypervisor. Includes kernel. Complete isolation. Startup: minutes. Size: GBs. Strong security boundary.
  • Container: Process isolation using Linux namespaces + cgroups. Shares host kernel. Startup: seconds. Size: MBs. Lighter isolation (same kernel vulnerabilities).

Choose containers for microservices, CI, stateless apps. Choose VMs for strong isolation, full OS control, legacy apps.

☸️ Kubernetes Questions

Explain the Kubernetes control plane components. Medium▼
  • kube-apiserver: Front door for all Kubernetes API calls. Validates and persists state to etcd.
  • etcd: Distributed key-value store. Stores ALL cluster state. Backup etcd = backup your cluster.
  • kube-scheduler: Assigns pods to nodes based on resource requests, affinity, taints/tolerations.
  • kube-controller-manager: Runs controllers: Deployment, ReplicaSet, Node, Endpoint controllers. Reconciliation loop.
  • cloud-controller-manager: Handles cloud-specific resources (LoadBalancer, PersistentVolume provisioning).
What happens when you run 'kubectl apply -f deployment.yaml'? Medium▼
  1. kubectl sends PATCH/POST to kube-apiserver
  2. apiserver authenticates (cert), authorizes (RBAC), validates schema, runs admission webhooks
  3. apiserver persists desired state to etcd
  4. Deployment controller detects change, creates/updates ReplicaSet
  5. ReplicaSet controller creates Pods in etcd (no node assigned yet)
  6. kube-scheduler finds suitable nodes (checks resources, affinity, taints)
  7. kubelet on the assigned node pulls image, starts container, reports status
What is the difference between Deployment, StatefulSet, and DaemonSet? Medium▼
  • Deployment: Stateless apps. Pods are interchangeable. Rolling updates, rollbacks. Use for web servers, APIs.
  • StatefulSet: Stateful apps. Pods have stable network identity (pod-0, pod-1) and persistent storage. Ordered, graceful deployment/deletion. Use for databases (MySQL, Kafka, Elasticsearch).
  • DaemonSet: Runs exactly one pod per node (or selected nodes). Use for monitoring agents, log collectors, network plugins.
A pod is stuck in Pending state. How do you troubleshoot? Hard▼
kubectl describe pod <pod-name> -n <namespace>  # look at Events section

Common causes from Events:

  • Insufficient CPU/memory: Nodes don't have enough resources. Check: kubectl describe nodes | grep -A5 Allocatable
  • No matching node (affinity/taint): Add toleration or fix nodeSelector
  • PVC not bound: Check kubectl get pvc — storage class not provisioning?
  • Image pull failure: Wrong tag, private registry missing imagePullSecret
  • No nodes at all: Check node count, cluster autoscaler logs
How does Kubernetes RBAC work? Give an example. Medium▼

RBAC grants permissions via: Subject (who) + Role (what permissions) + RoleBinding (binding).

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: myapp
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
---
kind: RoleBinding
metadata:
  name: read-pods
  namespace: myapp
subjects:
- kind: ServiceAccount
  name: myapp-sa
roleRef:
  kind: Role
  name: pod-reader

What is the difference between Security Groups and NACLs? Easy▼
  • Security Group: Stateful (return traffic automatic). Allow-only rules. Instance level (ENI). Applied per resource.
  • NACL: Stateless (must allow both directions explicitly). Allow AND Deny. Subnet level. Rules numbered by priority.

Use SGs for primary security control. Use NACLs to block specific IPs/ranges at subnet level (e.g., block known malicious IP ranges).

You need a Lambda function to access RDS in a private VPC. How do you set this up? Medium▼
  1. Configure Lambda to run inside the VPC: assign subnets (private) and security group
  2. Lambda creates ENIs in those subnets — needs sufficient IP addresses
  3. RDS security group must allow inbound from Lambda's security group
  4. Use RDS Proxy (recommended): reduces connection overhead, handles connection pooling, IAM auth
  5. Store DB credentials in Secrets Manager, reference in Lambda env vars
  6. Lambda execution role needs secretsmanager:GetSecretValue permission
Explain DynamoDB's partition key design and the hot partition problem. Hard▼

DynamoDB distributes data across partitions based on partition key hash. Each partition handles up to 3,000 RCUs and 1,000 WCUs.

Hot partition: When most traffic goes to one partition key (e.g., using date as PK — all writes hit today's partition). Symptoms: ProvisionedThroughputExceededException even with seemingly enough capacity.

Solutions:

  • High-cardinality PK (UUID, userId) — distributes evenly
  • Write sharding: append random suffix 0-N to PK, combine in GSI for reads
  • On-demand capacity mode (auto-scales per partition)
  • DAX for read-heavy patterns (cache in front of DynamoDB)

🔒 Security Questions

What is OWASP and what are the top 3 web vulnerabilities? Easy▼

OWASP (Open Worldwide Application Security Project) — non-profit foundation maintaining the OWASP Top 10, a standard awareness document for web security risks.

Top 3 (2021):

  1. Broken Access Control: Users can act beyond their intended permissions (IDOR, privilege escalation, force browsing)
  2. Cryptographic Failures: Sensitive data exposed due to weak/missing encryption (HTTP not HTTPS, MD5 passwords)
  3. Injection: SQLi, LDAP injection, OS command injection — untrusted data sent as part of a query/command
Explain how HTTPS/TLS works step by step. Medium▼
  1. Client sends ClientHello (TLS version, cipher suites, random)
  2. Server sends ServerHello + Certificate (public key + CA signature)
  3. Client verifies certificate against trusted CAs in trust store
  4. Key exchange: ECDHE (TLS 1.3) — both sides compute shared secret
  5. Session keys derived (symmetric AES-256)
  6. Server sends Finished, encrypted with session key
  7. Client sends Finished
  8. Application data encrypted with session keys

TLS 1.3 achieves this in 1 RTT (vs 2 RTT for TLS 1.2). All TLS 1.3 cipher suites provide Perfect Forward Secrecy.

How would you respond to discovering credentials were leaked in a public GitHub repo? Hard▼
  1. Immediate (within minutes): Revoke/rotate the leaked credentials immediately — assume they've already been used
  2. Audit: Check CloudTrail/audit logs for any API calls using the leaked key (time since exposure)
  3. Remove from Git: Use BFG Repo Cleaner or git filter-repo to purge from history (also need to force-push all branches, notify all contributors)
  4. Notify: Security team, management, potentially customers if their data was at risk
  5. Root cause: Why were credentials in code? Add pre-commit hooks (truffleHog, git-secrets), secret scanning in CI, educate team
  6. Prevent recurrence: Use AWS IAM Roles/OIDC, environment variables, secret managers — never hardcode credentials

🐍 Python Questions

What is the GIL and how does it affect Python concurrency? Hard▼

The GIL (Global Interpreter Lock) is a mutex in CPython that ensures only one thread executes Python bytecode at a time. This means multi-threading cannot achieve true CPU parallelism in CPython.

When GIL doesn't matter: I/O-bound tasks — GIL is released during I/O, so threading works well for web scraping, API calls, file I/O.

When GIL is a problem: CPU-bound tasks (video processing, ML inference, computation).

Workarounds: multiprocessing (separate processes, no shared GIL), asyncio (single-thread cooperative concurrency for I/O), C extensions (NumPy releases GIL), or use PyPy.

Explain Python decorators with a real-world example. Medium▼
import functools, time

def timer(func):
    @functools.wraps(func)  # preserves __name__, __doc__
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f}s")
        return result
    return wrapper

@timer
def slow_query(n):
    return sum(range(n))

slow_query(10_000_000)  # "slow_query took 0.2341s"

Real-world uses: @login_required (Django), @cache, @retry, @rate_limit, @validate_input.

What is the difference between 'is' and '==' in Python? Easy▼
  • == checks value equality (calls __eq__ method)
  • is checks identity — same object in memory (same id())
a = [1, 2, 3]
b = [1, 2, 3]
a == b   # True (same values)
a is b   # False (different objects)

x = None
x is None    # True (correct way to check for None)
x == None    # True but bad practice — use 'is'

Small integers (-5 to 256) and short strings are interned — is may return True unexpectedly. Never rely on is for value comparison.

What are Python generators and when should you use them? Medium▼

Generators produce values one at a time using yield — they're lazy (evaluate on demand). They maintain state between yields and use O(1) memory regardless of data size.

def read_large_file(filename):
    with open(filename) as f:
        for line in f:
            yield line.strip()  # one line at a time

# Generator expression
squares = (x**2 for x in range(1_000_000))  # no list created
total = sum(squares)  # iterates lazily

# Infinite sequence
def fibonacci():
    a, b = 0, 1
    while True:
        yield a
        a, b = b, a + b

🗄️ Databases Questions

What is the difference between SQL and NoSQL databases? When to use each? Easy▼
  • SQL (Relational): Structured schema, ACID transactions, complex queries with JOINs. Use for: financial systems, e-commerce, CRM, anywhere data consistency is critical.
  • NoSQL: Flexible schema, horizontal scaling, various data models (document, key-value, column, graph). Use for: high-scale reads/writes, variable schema, specific access patterns (DynamoDB, MongoDB, Redis).

Don't choose NoSQL just for "scale" — modern SQL (PostgreSQL, Aurora) scales extremely well. Choose based on access patterns, not hype.

Explain database indexes — types and trade-offs. Medium▼

B-Tree index (default): Balanced tree. Good for =, <, >, BETWEEN, ORDER BY. Most queries.

Hash index: O(1) equality lookups only. No range queries. PostgreSQL supports.

Composite index: Multi-column. Column order matters! Index (a, b) helps queries on (a), (a, b) but NOT (b) alone.

Partial index: Index only rows matching a condition. Smaller, faster.

Trade-offs: Indexes speed up reads but slow down writes (INSERT/UPDATE/DELETE must update all indexes). Every index uses disk space. Don't over-index — target your actual query patterns.

What is a database transaction and what does ACID guarantee? Medium▼
  • Atomicity: All operations in a transaction succeed or all are rolled back. No partial updates.
  • Consistency: Transaction brings DB from one valid state to another. All constraints (FK, unique, check) satisfied.
  • Isolation: Concurrent transactions behave as if sequential. Isolation levels: Read Uncommitted → Read Committed → Repeatable Read → Serializable (stronger = slower).
  • Durability: Committed transactions survive crashes. Write-ahead log (WAL) ensures this.
How does database replication work and what is replication lag? Hard▼

Synchronous replication: Primary waits for replica acknowledgment before confirming write. Zero replication lag. Slower writes. Used by RDS Multi-AZ.

Asynchronous replication: Primary doesn't wait. Replica may lag by seconds/minutes. Faster writes but potential data loss on failover. Used by Read Replicas.

Replication lag causes: Heavy write load, large transactions, network issues, slow replica hardware.

Impact: Reading from a lagging replica may return stale data. Application must handle this: direct writes to primary, reads to replica with acceptable lag, or use strongly consistent reads when needed.

🏛️ System Design Questions

How would you design a URL shortener like bit.ly? Medium▼

Requirements: Shorten URL, redirect, analytics (optional), 100M URLs, 10B redirects/day.

Core design:

  • Generate short code: Base62 encode a unique ID (counter or UUID). 6 chars = 62^6 ≈ 56B URLs.
  • Database: short_code (PK) → original_url + created_at + user_id
  • Cache: Redis with short_code → URL (99%+ cache hit rate expected)
  • Redirect: HTTP 301 (permanent, browser caches) or 302 (temporary, tracks clicks)
  • Scale reads: Redis cluster, CDN for redirect responses
Design a distributed rate limiter for an API. Hard▼

Algorithms:

  • Token Bucket: Bucket refills at fixed rate. Allows bursting up to bucket size. Simple with Redis INCR + TTL.
  • Sliding Window Log: Store timestamps of requests in sorted set. Count requests in last window. Accurate but memory-heavy.
  • Fixed Window Counter: Count per time window. Simple but allows 2x burst at window boundary.

Redis implementation:

-- Lua script (atomic)
local count = redis.call('INCR', key)
if count == 1 then
  redis.call('EXPIRE', key, window_seconds)
end
return count

Scale: Redis Cluster for multi-node. Rate limit by user_id + API_key. Return Retry-After header on 429.

What is the CAP theorem and how does it apply to real systems? Hard▼

CAP theorem: A distributed system can only guarantee 2 of 3: Consistency (all nodes see same data), Availability (every request gets a response), Partition Tolerance (system works despite network partitions).

Network partitions are unavoidable in distributed systems, so you must choose between C and A during a partition:

  • CP systems: Reject requests during partition to maintain consistency. HBase, ZooKeeper, etcd.
  • AP systems: Serve potentially stale data to remain available. Cassandra, CouchDB, DynamoDB (eventually consistent).

Real systems: Most cloud databases are tunable (DynamoDB: eventual vs strong consistency per read, Cassandra: consistency level per operation).

💼 Behavioral Questions

Tell me about a time you had to deal with a critical production incident. Medium▼

STAR method (Situation, Task, Action, Result):

Structure your answer to cover: What was the incident? What was your role? What steps did you take to diagnose and fix it? What was the outcome? What did you learn?

Key points to hit: How you stayed calm, how you communicated with stakeholders, the technical troubleshooting steps, how you prevented recurrence (post-mortem, monitoring improvements, runbooks).

Example framework: "Our payment service was returning 500s affecting X% of users. I [action]. After Y minutes we resolved it by [fix]. We implemented Z to prevent recurrence."

How do you handle disagreements with teammates about technical decisions? Easy▼
  • Listen to understand their perspective first — they may have context you don't
  • Focus on data and technical merit, not personalities
  • Propose a proof-of-concept or benchmark to resolve factual disagreements
  • Document the trade-offs of both approaches objectively
  • If still unresolved, escalate to tech lead or use RFC process
  • Once a decision is made, commit fully regardless of who "won"
  • Learn from outcomes — note if your or their approach proved better for future decisions
How do you prioritize when you have multiple urgent tasks? Easy▼
  • Assess impact and urgency: Eisenhower Matrix — Is it urgent AND important?
  • Clarify with stakeholders: "If I can only do one of X or Y today, which matters more?"
  • Tackle blockers first: Unblock teammates before individual work
  • Time-box: Set time limits to avoid over-investing in one task
  • Communicate proactively: If something will slip, tell stakeholders early with a new ETA
  • Learn patterns: If urgent tasks are always popping up, address the root cause (better planning, on-call rotation)
