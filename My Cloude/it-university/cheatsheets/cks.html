Network Policies (Default Deny Pattern)

System Hardening 15%
Step 1 — Default Deny All Ingress
apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: default-deny-ingress namespace: production spec: podSelector: {} # selects ALL pods in namespace policyTypes: - Ingress # deny all ingress, egress still allowed
Default Deny All (Ingress + Egress)
apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: default-deny-all spec: podSelector: {} policyTypes: - Ingress - Egress # blocks DNS too — add DNS allow below!
Allow DNS Egress (required if denying all egress)
apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: allow-dns spec: podSelector: {} policyTypes: - Egress egress: - ports: - port: 53 protocol: UDP - port: 53 protocol: TCP
Allow Specific Pod-to-Pod Traffic
spec: podSelector: matchLabels: app: backend policyTypes: - Ingress ingress: - from: - podSelector: # same namespace matchLabels: app: frontend - namespaceSelector: # from specific namespace matchLabels: kubernetes.io/metadata.name: frontend-ns ports: - port: 8080
Allow Egress to Specific CIDR
egress: - to: - ipBlock: cidr: 10.0.0.0/8 except: - 10.0.0.0/24 # exclude this subnet ports: - port: 5432
Test NetworkPolicy Connectivity
k exec -it test-pod -- wget -qO- http://backend-svc:8080 k exec -it test-pod -- nc -zv backend-svc 8080 k exec -it test-pod -- curl -v http://backend-svc --max-time 5
Note: NetworkPolicy requires a CNI plugin that supports it (Calico, Cilium, Weave). Flannel does NOT enforce NetworkPolicies.

Pod Security

Minimize Microservice Vulnerabilities 20%
Pod Security Admission (PSA) — Label Namespace
k label namespace myns \ pod-security.kubernetes.io/enforce=restricted \ pod-security.kubernetes.io/enforce-version=v1.29 \ pod-security.kubernetes.io/warn=restricted \ pod-security.kubernetes.io/audit=restricted
PSA Levels
privilegedNo restrictions (default kube-system)
baselinePrevents known privilege escalations
restrictedHeavily restricted, best practices
Restricted Profile securityContext
securityContext: runAsNonRoot: true runAsUser: 1000 # must be non-zero seccompProfile: type: RuntimeDefault # or Localhost allowPrivilegeEscalation: false capabilities: drop: - ALL # drop all capabilities add: - NET_BIND_SERVICE # only add if needed readOnlyRootFilesystem: true
OPA Gatekeeper — ConstraintTemplate
apiVersion: templates.gatekeeper.sh/v1 kind: ConstraintTemplate metadata: name: k8srequiredlabels spec: crd: spec: names: kind: K8sRequiredLabels targets: - target: admission.k8s.gatekeeper.sh rego: | package k8srequiredlabels violation[{"msg": msg}] { provided := {label | input.review.object.metadata.labels[label]} required := {label | label := input.parameters.labels[_]} missing := required - provided count(missing) > 0 msg := sprintf("missing labels: %v", [missing]) }
OPA Gatekeeper — Constraint (enforces template)
apiVersion: constraints.gatekeeper.sh/v1beta1 kind: K8sRequiredLabels spec: match: kinds: - apiGroups: [""] kinds: ["Pod"] parameters: labels: ["app", "env"]
OPA Gatekeeper uses two objects: ConstraintTemplate (Rego logic) + Constraint (applies the policy). Both must exist for enforcement.

Custom Rule — Detect Crypto Mining
- rule: Detect Crypto Mining desc: Detect if a container is attempting crypto mining condition: > spawned_process and container and (proc.name = "xmrig" or proc.cmdline contains "stratum+tcp") output: > Crypto mining detected (user=%user.name container=%container.name image=%container.image.repository proc=%proc.name cmd=%proc.cmdline) priority: CRITICAL tags: [cryptomining, mitre_execution]
Rule Components
conditionFalco expression (proc.name, container.id, fd.name, evt.type)

etcd Encryption at Rest

Cluster Setup 10%
Create EncryptionConfiguration
# /etc/kubernetes/enc/enc.yaml apiVersion: apiserver.config.k8s.io/v1 kind: EncryptionConfiguration resources: - resources: - secrets - configmaps # optional providers: - aescbc: # recommended provider keys: - name: key1 secret: c2VjcmV0aXNzZWNyZXQ= # base64 32-byte key - identity: {} # fallback for existing unencrypted data
Generate Encryption Key
# Generate a 32-byte base64 key head -c 32 /dev/urandom | base64
Enable in kube-apiserver
--encryption-provider-config=/etc/kubernetes/enc/enc.yaml
Re-encrypt Existing Secrets
# Force re-encrypt all existing secrets kubectl get secrets --all-namespaces -o json | kubectl replace -f -
Verify Encryption via etcdctl
ETCDCTL_API=3 etcdctl \ --endpoints=https://127.0.0.1:2379 \ --cacert=/etc/kubernetes/pki/etcd/ca.crt \ --cert=/etc/kubernetes/pki/etcd/server.crt \ --key=/etc/kubernetes/pki/etcd/server.key \ get /registry/secrets/default/mysecret | hexdump -C # Should see "k8s:enc:aescbc:v1:key1:" prefix in output
Encryption Providers
identityNo encryption (plaintext) — default
aescbcAES-CBC PKCS#7 — stable, recommended
aesgcmAES-GCM — faster, rotate keys often!
secretboxXSalsa20 + Poly1305 — good alternative
kmsExternal KMS (AWS KMS, Vault) — most secure
The identity: {} provider must come LAST in the list — it acts as fallback to read unencrypted data that was stored before encryption was enabled.
Order matters: first provider is used for writing. Put aescbc first so new secrets are encrypted. identity last for reading old data.

Create CSR and Sign with Cluster CA
# Create CSR for new cert openssl req -new -key mykey.key -out myrequest.csr \ -subj "/CN=my-user/O=my-group" # Sign with cluster CA openssl x509 -req -in myrequest.csr \ -CA /etc/kubernetes/pki/ca.crt \ -CAkey /etc/kubernetes/pki/ca.key \ -CAcreateserial -out mycert.crt -days 365

# /var/lib/kubelet/config.yaml serverTLSBootstrap: true # auto-rotate server certs # Approve pending CSRs kubectl get csr kubectl certificate approve <csr-name> kubectl certificate deny <csr-name>

IstioPeerAuthentication (STRICT/PERMISSIVE/DISABLE), DestinationRule

Audit Policy & Log Analysis

Monitoring, Logging & Runtime Security 20%
Audit Levels
NoneDo not log this event
MetadataLog request metadata (user, time, resource, verb)
RequestLog metadata + request body
RequestResponseLog metadata + request + response body
Audit Policy Example
apiVersion: audit.k8s.io/v1 kind: Policy omitStages: - RequestReceived # don't log this stage rules: # Don't log read-only requests to non-sensitive resources - level: None verbs: ["get", "watch", "list"] resources: - group: "" resources: ["endpoints", "services", "configmaps"] # Don't log requests from system components - level: None users: - system:kube-proxy - system:apiserver verbs: ["watch"] # Log secrets at Metadata level (don't log content) - level: Metadata resources: - group: "" resources: ["secrets", "configmaps", "tokenreviews"] # Log pod changes at Request level - level: Request verbs: ["create", "update", "patch", "delete"] resources: - group: "" resources: ["pods"] # Log everything else at Metadata - level: Metadata omitStages: - RequestReceived
Audit Log Analysis
# Watch audit log in real time tail -f /var/log/kubernetes/audit.log | jq . # Find all operations on secrets cat /var/log/kubernetes/audit.log | jq 'select(.objectRef.resource=="secrets")' # Find failed auth attempts (403 / 401) cat /var/log/kubernetes/audit.log | \ jq 'select(.responseStatus.code == 403 or .responseStatus.code == 401)' # Find who deleted pods cat /var/log/kubernetes/audit.log | \ jq 'select(.verb=="delete" and .objectRef.resource=="pods") | {user:.user.username, pod:.objectRef.name}'
Suspicious Audit Patterns
system:anonymousAnonymous requests — block immediately
Unexpected secret accessSA accessing secrets from wrong namespace
exec/attachShell access to sensitive system pods
RBAC escalationAttempts to create ClusterRoleBindings
Unusual namespaceAccess to kube-system from app accounts
Audit policy rules are evaluated in ORDER — first match wins. Put None rules at top to reduce noise, catch-all Metadata at the bottom.

Secure Ingress & Service Exposure

Cluster Setup 10%
Create TLS Secret for Ingress
# Create TLS secret from cert files kubectl create secret tls my-tls-secret \ --cert=tls.crt \ --key=tls.key \ -n myns
Ingress with TLS Termination
apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: my-ingress annotations: nginx.ingress.kubernetes.io/ssl-redirect: "true" nginx.ingress.kubernetes.io/force-ssl-redirect: "true" spec: tls: - hosts: - myapp.example.com secretName: my-tls-secret rules: - host: myapp.example.com http: paths: - path: / pathType: Prefix backend: service: name: myapp-svc port: { number: 80 }
NodePort vs LoadBalancer Risk
ClusterIPInternal only — most secure default
NodePortExposes on all nodes — restrict with NetworkPolicy
LoadBalancerPublic IP — use only with cloud firewall rules
ExternalNameDNS alias — verify target is trusted
Avoid NodePort in production. Prefer Ingress with TLS for HTTP services. For non-HTTP, use LoadBalancer with cloud security groups.

RuntimeClass & Sandboxing

Minimize Microservice Vulnerabilities 20%
RuntimeClass Definition
apiVersion: node.k8s.io/v1 kind: RuntimeClass metadata: name: gvisor handler: runsc # gVisor handler --- apiVersion: node.k8s.io/v1 kind: RuntimeClass metadata: name: kata handler: kata-containers # Kata Containers handler
Use RuntimeClass in Pod
spec: runtimeClassName: gvisor # use sandboxed runtime containers: - name: myapp image: myapp:1.0
Sandbox Runtimes Comparison
gVisor (runsc)User-space kernel, intercepts syscalls
Kata ContainersLightweight VM per pod, full isolation
runcDefault OCI runtime, shares host kernel
Container Runtime Interface
# Check running containers crictl ps crictl pods crictl inspect <container-id> crictl logs <container-id> # Pull and verify image crictl pull nginx:latest crictl images
Use gVisor for untrusted workloads. It adds overhead (~10-30%) but isolates container syscalls from the host kernel — container escape resistance is much higher.
