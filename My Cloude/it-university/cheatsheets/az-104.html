RBAC — Role-Based Access Control

Scope Hierarchy (broadest → narrowest)
Management Group → Subscription → Resource Group → Resource
Roles assigned at higher scope are inherited downward. Deny assignments (from Azure Blueprints/PIM) override Allow.
Built-in Roles
OwnerFull access + assign roles
ContributorFull access, cannot assign roles
ReaderRead-only everything
User Access AdminManage access only, no resource ops
Assignment CLI
az role assignment create \ --assignee john@domain.com \ --role "Contributor" \ --scope /subscriptions/{sub}/resourceGroups/{rg} az role assignment list \ --assignee john@domain.com az role assignment delete \ --assignee john@domain.com \ --role "Contributor" \ --scope /subscriptions/{sub}/resourceGroups/{rg}
Custom Role JSON
{ "Name": "VM Operator", "Actions": ["Microsoft.Compute/*/read", "Microsoft.Compute/virtualMachines/start/action"], "NotActions": [], "DataActions": [], "AssignableScopes": ["/subscriptions/{sub}"] }
az role definition create \ --role-definition role.json

NSG — Network Security Groups

NSG CLI
az network nsg create \ --resource-group myRG --name myNSG az network nsg rule create \ --resource-group myRG \ --nsg-name myNSG \ --name AllowHTTP \ --priority 100 \ --direction Inbound \ --access Allow \ --protocol Tcp \ --source-address-prefix '*' \ --destination-port-range 80 az network nsg rule list \ --resource-group myRG \ --nsg-name myNSG -o table
Priority Rules
Lower number = higher priority. Valid range: 100–4096. Rules are processed in order; first match wins. Ranges 65000–65500 are reserved defaults.
Default Inbound Rules (cannot delete)

65001AllowAzureLoadBalancerInBoundAllow
65500DenyAllInBoundDeny
Default Outbound Rules
PriorityRule NameAction
65000AllowVnetOutBoundAllow

65500DenyAllOutBoundDeny
Application Security Groups (ASG)
ASGs let you group VMs logically (e.g. "WebServers", "DBServers") and use them as source/destination in NSG rules — no need to manage IP addresses.

VNet Peering & UDR

VNet Peering
az network vnet peering create \ --resource-group myRG \ --name peer1to2 \ --vnet-name vnet1 \ --remote-vnet vnet2 \ --allow-vnet-access # Must create reciprocal peering az network vnet peering create \ --resource-group myRG \ --name peer2to1 \ --vnet-name vnet2 \ --remote-vnet vnet1 \ --allow-vnet-access
Non-transitive: VNet A peers B, B peers C — A cannot reach C. Must add A-C peering explicitly or use hub-spoke with Azure Firewall/VPN Gateway (allow gateway transit).

Use Remote GatewaysSpoke side — use hub gateway
Global peeringAcross regions; higher latency, extra cost
