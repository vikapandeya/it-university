Threat Detection Services

Domain 1 — 14%
Amazon GuardDuty
  • Managed threat detection; analyzes VPC Flow Logs, CloudTrail, DNS logs, S3 data events
  • Findings severity: HIGH / MEDIUM / LOW
  • Multi-account via AWS Organizations (delegated admin)
  • 30-day free trial per account per region
  • Add-ons: Malware Protection (EBS volume scanning), EKS Audit Log, RDS Login Activity
  • Does NOT prevent — detection only; pair with EventBridge + Lambda for response
Amazon Detective
  • Investigate why a finding occurred; deep-dive after GuardDuty alert
  • Visualizes behavior graphs using VPC Flow Logs, CloudTrail, GuardDuty findings
  • Retains 12 months of data; cannot query older data
  • NOT for detection — for post-detection investigation
Amazon Inspector
  • Automated vulnerability scanning: EC2 (needs SSM agent), ECR images, Lambda functions
  • CVE detection, CVSS scores; continuous scanning (not on-demand)
  • Replaces legacy Inspector Classic; push findings to Security Hub
AWS Security Hub
  • Aggregates findings from GuardDuty, Inspector, Macie, IAM Analyzer, Firewall Manager
  • CSPM (Cloud Security Posture Management); CIS benchmark checks
  • Custom insights; cross-account & cross-region aggregation
  • ASFF (Amazon Security Finding Format) — standard format for all findings
  • Integrates with 3rd-party tools (Splunk, PagerDuty, etc.)
Amazon Macie
  • ML-powered S3 data classification; detects PII, PHI, financial data
  • Sensitive data discovery jobs; automated & on-demand
  • Findings → Security Hub; multi-account support via Organizations
  • Classifies up to 1 GB per S3 object for PII
Key distinctions: GuardDuty = detect threats | Detective = investigate findings | Inspector = scan vulnerabilities | Macie = classify S3 data | Security Hub = aggregate & correlate
GuardDuty Security-Hub Detective Inspector Macie

Security Logging & Monitoring

Domain 2 — 18%
AWS CloudTrail
  • API audit log — who did what, when, from where
  • Management events: enabled by default, free in Event History (90 days)
  • Data events: S3 object-level ops (GetObject, PutObject), Lambda invoke — extra cost, disabled by default
  • Insights: detects unusual API call rates or error rates
  • Multi-region trail; logs to S3 + optionally CloudWatch Logs or EventBridge
  • Log file integrity: SHA-256 hash + RSA-SHA-256 digital signature
  • CloudTrail Lake: immutable event data store, SQL queries, up to 7-year retention
VPC Flow Logs
  • Captures network traffic metadata (not payload/content)
  • Levels: ENI, Subnet, VPC; published to S3 or CloudWatch Logs
  • NOT captured: DNS queries to Route 53 resolver, DHCP traffic, Windows license activation, EC2 metadata (169.254.x.x), traffic to reserved IPs
  • Use Traffic Mirroring for packet-level capture (payload)
Amazon CloudWatch Logs
  • Log groups → log streams; Metric Filters create CloudWatch metrics from log patterns
  • Insights: query language for ad-hoc log analysis
  • Subscription filters: stream logs to Lambda, Firehose, OpenSearch
  • Retention: 1 day to 10 years (default: never expire)
AWS Config
  • Tracks configuration state changes of AWS resources over time
  • Rules: managed (AWS provided) or custom (Lambda-backed); evaluate compliance
  • Remediation: auto-remediation via SSM Automation documents
  • Aggregation across accounts/regions; Conformance Packs (bundle of rules)
  • NOT real-time — can have minutes of delay; NOT for API call logging (use CloudTrail)
Amazon EventBridge
  • Near real-time event routing; rules match patterns from AWS services
  • Targets: Lambda, SNS, SQS, Step Functions, ECS, Kinesis
  • CloudTrail events → EventBridge rules for automated response
  • Scheduled rules (cron/rate) for periodic security checks
Exam trap: CloudTrail does NOT log S3 GetObject / PutObject by default — must enable data events. Config tracks state, NOT API calls. For real-time response, use EventBridge (not Config).
CloudTrail VPC-Flow-Logs Config CloudWatch EventBridge

Identity & Access Management (Advanced)

Domain 4 — 16%
Policy Evaluation Logic
Evaluation OrderOrg SCP → Resource policy → IAM Permission Boundary → Identity policy → Session policy
Explicit DenyAlways wins over any Allow, in any policy layer
Same-account unionIdentity policy + Resource policy in same account = UNION (either allows = access)
Cross-accountBOTH resource policy + identity policy (or role) must allow
Permission Boundaries
  • Managed policy that sets maximum permissions for an IAM entity
  • Effective permissions = intersection of identity policy AND boundary
  • Does NOT grant permissions — only limits them
  • Useful for delegating admin without privilege escalation risk
SCPs (Service Control Policies)
  • Organization-level; restricts even root users of member accounts
  • Evaluated before any IAM policy; cannot grant permissions
  • Max size: 5,120 characters per policy
  • Attached to OU or account; inherited down the hierarchy
ABAC & Identity Center
ABACTag-based access control; aws:PrincipalTag, aws:ResourceTag condition keys; scalable for dynamic environments
Identity Center (SSO)Central access to AWS accounts + SaaS; AD/LDAP/SAML 2.0 IdP; Permission Sets; SCIM provisioning
IAM Access AnalyzerFinds externally shared resources (S3, roles, KMS, Lambda, SQS); validates policies; unused access analysis
Key Limits
Managed policies/entity10 managed policies per user or role
Inline policy size6,144 characters per inline policy
Permission boundary is NOT a grant — you still need an identity policy that allows the action. The boundary only caps the ceiling.
Permission-Boundaries SCPs ABAC Identity-Center Access-Analyzer

Data Protection — KMS & Encryption

Domain 5 — 18%
KMS Key Types
AWS Managed Keysaws/service prefix; auto-rotate every year; you cannot delete or disable
Customer Managed (CMK)You control rotation, deletion, key policy; 7–30 day deletion waiting period (default 30)
AWS Owned KeysMulti-tenant, fully AWS managed; no visibility or control
KMS Key Operations
EncryptDirectly encrypts up to 4 KB of data
GenerateDataKeyReturns plaintext DEK + encrypted DEK for envelope encryption
GenerateDataKeyWithoutPlaintextReturns only encrypted DEK; plaintext generated later in HSM
DecryptDecrypts ciphertext encrypted by KMS key
ReEncryptRe-encrypts data under a different CMK without exposing plaintext
Request rate5,500 – 30,000 req/sec (varies by region & key type)
Envelope Encryption
  • Data encrypted with Data Encryption Key (DEK)
  • DEK encrypted with CMK; stored alongside ciphertext
  • Enables encrypting data >4 KB (CMK can only directly encrypt 4 KB)
  • DEK decryption requires KMS call; plaintext DEK used only in memory
KMS Key Policies
  • Must explicitly grant access — even to account root
  • Key policy + IAM policy BOTH must allow for IAM entities
  • Key grants: programmatic, temporary delegation (no policy change needed)
CloudHSM
  • Single-tenant dedicated HSM; FIPS 140-2 Level 3 validated
  • You control keys — AWS cannot access key material
  • Clustered for HA; deploy across AZs
  • Use cases: SSL offload, Oracle TDE, custom key store for KMS, PKI
  • More expensive than KMS; required when FIPS Level 3 needed
Secrets Manager vs Parameter Store
Secrets ManagerStore, rotate, audit secrets; native rotation for RDS/Redshift/DocumentDB; Lambda for custom rotation; versioning: AWSCURRENT / AWSPENDING / AWSPREVIOUS
Parameter StoreCheaper; Standard (free) / Advanced; SecureString uses KMS; no auto-rotation
Rotation Lambda timeout30 seconds
S3 Encryption Options
SSE-S3AES-256; AWS manages keys; header: x-amz-server-side-encryption: AES256
SSE-KMSCMK; audit trail in CloudTrail; cross-account support; header: aws:kms
SSE-CCustomer provides key with each request; AWS does NOT store key
CSEClient-side encryption before upload; full key control
ACM (Certificate Manager)
  • SSL/TLS certs for ALB, CloudFront, API GW; auto-renewal
  • ACM Private CA: internal PKI for private certificates
  • Cannot export ACM public certs (use for AWS services only)
CloudHSM vs KMS: CloudHSM = you own keys, FIPS Level 3, expensive, single-tenant. KMS = AWS manages HSMs, FIPS Level 2 (some Level 3), cheaper, natively integrated with 100+ AWS services.
KMS CloudHSM Secrets-Manager ACM envelope-encryption SSE-KMS

Incident Response

Domain 1 (Advanced) — Playbooks & Containment
Compromised IAM Credentials
  • 1. Disable/delete compromised access keys immediately
  • 2. Revoke active sessions: attach inline deny policy with aws:TokenIssueTime condition
  • 3. Check CloudTrail for all API calls made with those credentials
  • 4. Rotate all keys; check for newly created IAM users / privilege escalation
  • 5. Notify AWS Support if root credentials compromised
Compromised EC2 Instance
  • 1. Isolate: change Security Group to deny-all (no inbound/outbound)
  • 2. Preserve evidence: create EBS snapshot, capture memory dump if possible
  • 3. Enable termination protection to prevent accidental deletion
  • 4. Analyze with forensics AMI (separate VPC, no internet)
  • 5. Check instance metadata for IAM role credential exfiltration (SSRF → IMDS)
  • Use IMDSv2 (session-oriented) to prevent SSRF-based IMDS attacks
Exposed S3 Bucket
  • Enable S3 Block Public Access at account level immediately
  • Review bucket policy + ACL for public grants
  • Enable S3 Object Ownership (disable ACLs) for centralized control
  • Use Macie to identify what sensitive data was potentially exposed
  • Check CloudTrail S3 data events for unauthorized access
Never delete compromised resources immediately — preserve evidence first (snapshots, logs). Disable & isolate, then investigate, then remediate.
GuardDuty Finding Response
  • High severity → EventBridge rule → Lambda → isolate + notify
  • Use Step Functions for multi-step automated playbooks
  • GuardDuty finding types: UnauthorizedAccess, CryptoCurrency, Backdoor, Recon, Trojan, Policy violations
  • Suppress findings using suppression rules for known-good activity
Automated Remediation Patterns
Config + SSMConfig rule detects drift → SSM Automation document remediates
Security Hub + EventBridgeSecurity Hub finding → EventBridge → Lambda auto-remediation
Incident ManagerAWS Systems Manager Incident Manager for runbooks & on-call escalation
NIST Incident Response Framework
1. PreparationPolicies, training, tools, runbooks in place before incident
2. DetectionGuardDuty, Security Hub, CloudWatch alarms, Macie findings
3. AnalysisDetective, CloudTrail Lake, VPC Flow Logs investigation
4. ContainmentIsolate EC2, revoke credentials, block IPs in WAF/NACLs
5. EradicationRemove malware, patch vulnerability, rotate all secrets
6. RecoveryRestore from clean backup/snapshot; verify integrity
7. Post-IncidentLessons learned, update runbooks, improve detection rules
incident-response forensics containment GuardDuty-remediation CloudTrail-investigation

Exam Tips — Critical Decision Trees

High-yield scenarios
Service Selection Keywords
"Detect threats"Amazon GuardDuty (NOT Inspector or Security Hub)
"Investigate / why did this happen"Amazon Detective (post-detection analysis)
"Vulnerability scan / CVE"Amazon Inspector (EC2, ECR, Lambda)
"PII / sensitive data in S3"Amazon Macie (ML-powered classification)
"Aggregate findings / CSPM"AWS Security Hub (central dashboard)
"API who did what"AWS CloudTrail (audit trail)
"Config drift / compliance"AWS Config + Config Rules
"Real-time event response"Amazon EventBridge + Lambda
Multi-Account Security
  • Multi-account governance: Security Hub + Organizations + Firewall Manager + Config Aggregator
  • Centralized logging: dedicated Log Archive account in AWS Control Tower
  • SCPs at Organization root for blanket restrictions (e.g., block GuardDuty disable)
  • Delegated administrator accounts for Security Hub, GuardDuty, Macie, Inspector
Root User Compromise
  • Immediately: change root password + enable/change MFA
  • Check CloudTrail for all root API calls
  • Contact AWS Support — they can lock root if needed
  • Review all IAM users/roles created during compromise window
Data Exfiltration Prevention
  • S3 VPC endpoint + endpoint policy denying non-org principals
  • SCP denying s3:GetObject requests outside organization
  • S3 Block Public Access at account level
  • GuardDuty detects S3 data exfiltration patterns
Encryption Edge Cases
Encrypt existing unencrypted RDSCannot encrypt in-place; take snapshot → copy snapshot WITH encryption → restore from encrypted snapshot
Encrypt at rest automatically (RDS)Enable SSE with KMS at DB creation; cannot add encryption to existing unencrypted DB
Cross-account S3 accessSame org: bucket policy alone works. External account: BOTH bucket policy AND requester's IAM policy must allow
CloudHSM requirementFIPS Level 3 required, Oracle TDE, full key ownership, custom key store
IAM Policy Traps
  • Permission boundary is NOT a grant — it only limits the ceiling; identity policy still needed
  • SCP cannot grant permissions — only restrict; even if SCP allows, IAM policy must also allow
  • Resource-based policy in same account: union applies (either allows = access granted)
  • Cross-account: both policies MUST allow — union does NOT apply across accounts
CloudTrail Gotchas
  • CloudTrail is eventually consistent for recent events — don't rely on it for real-time
  • For real-time response, create EventBridge rules on CloudTrail events
  • S3 GetObject / PutObject NOT logged by default — enable data events
  • Management events free in Event History (90 days); trail to S3 for longer retention
Network Security Traps
  • NACLs are stateless — must explicitly allow return (ephemeral) traffic ports 1024-65535
  • Security Groups are stateful — return traffic automatically allowed
  • Default NACL allows all; custom NACL denies all until rules added
  • WAF doesn't protect against L3/L4 DDoS — need Shield Standard/Advanced
IMDSv2 (session-oriented metadata service) prevents SSRF attacks from reaching instance metadata. Always enforce IMDSv2 via instance metadata options or SCP.

Network & VPC Security Deep Dive

Domain 3 Advanced
VPC Endpoint Security
Interface EndpointENI in subnet; uses private IP; powered by AWS PrivateLink; supports endpoint policies
Gateway EndpointRoute table entry; S3 and DynamoDB only; no additional charge; endpoint policies supported
Endpoint PolicyResource policy on the endpoint; restricts access to specific S3 buckets or DynamoDB tables
Prevent data exfilS3 bucket policy: deny if aws:sourceVpce != your endpoint ID; forces traffic via VPC endpoint
DNS Firewall (Route 53 Resolver)
  • Filters DNS queries from VPC; block domains resolving to known malicious IPs
  • Managed domain lists (AWS managed + custom); BLOCK, ALLOW, ALERT actions
  • Prevents DNS exfiltration (data tunneled over DNS)
  • Managed by Firewall Manager across org
AWS PrivateLink vs VPC Peering
PrivateLinkExpose specific service privately; no route table changes; no IP overlap issues; one-directional
VPC PeeringFull VPC-to-VPC connectivity; no transitive routing; requires non-overlapping CIDRs
Transit GWHub-spoke model; supports transitive routing; VPN + Direct Connect; RAM for cross-account sharing
Network Firewall Architecture
  • Deploy in dedicated firewall subnets in each AZ
  • Route traffic through firewall via route tables (Gateway Load Balancer or inline)
  • Stateful rules: 5-tuple matching + Suricata IPS signature rules
  • FQDN filtering: block by domain name even without IP knowledge
  • TLS inspection: decrypt/inspect/re-encrypt HTTPS traffic
VPC-Endpoints DNS-Firewall PrivateLink Network-Firewall

Container & Serverless Security

Domain 3 + Domain 5
ECS / EKS Security
ECS Task RoleIAM role for the task (fine-grained); use task role, NOT instance role, for least privilege
ECS FargateNo EC2 management; tasks isolated per VPC; no SSH; ephemeral storage encrypted
EKS RBACaws-auth ConfigMap maps IAM to Kubernetes RBAC; IAM Authenticator
ECR ScanningInspector scans ECR images on push for CVEs; enable enhanced scanning
ECR PermissionsResource-based policy (repo policy) for cross-account access; private registry
Lambda Security
  • Execution role: IAM role assumed by Lambda; apply least privilege
  • Resource policy: controls who can invoke the function (cross-account invocations)
  • Lambda in VPC: function deployed in private subnet; needs NAT for internet access
  • Inspector scans Lambda code for vulnerabilities (CVE in dependencies)
  • Secrets Manager / Parameter Store for secrets — never hardcode credentials in code or env vars
  • Lambda function URL: enable auth (AWS_IAM) to prevent public invocation
Secrets in Applications
Anti-patternHardcoded creds in code, env vars, or container images — always detected by Macie/Inspector
Best practiceSecrets Manager or SSM Parameter Store (SecureString) + IAM role access only
CodeBuild / CI/CDUse IAM role for CodeBuild; inject secrets at build time via Secrets Manager — never in buildspec.yml
GuardDuty has dedicated protection for EKS (Audit Log Monitoring, Runtime Monitoring) and ECS (Runtime) — enable separately from base GuardDuty.
ECS-Task-Role Lambda-Execution-Role ECR-Scanning Secrets-Manager

Advanced Key Management & PKI

Domain 5 Advanced
KMS Custom Key Stores
  • CloudHSM-backed custom key store: KMS operations performed in your CloudHSM cluster
  • Key material generated AND used inside your HSM — AWS never sees it
  • Use when regulatory requirements mandate customer-controlled HSM + KMS API convenience
  • Higher latency than standard KMS; CloudHSM cluster must be dedicated to KMS
KMS Key Rotation & Import
Automatic rotationCMK: optional annual rotation; AWS managed: automatic annual; old key versions retained for decrypt
Manual rotationCreate new CMK → update all aliases → re-encrypt data; use for asymmetric keys (not auto-rotatable)
Import key materialGenerate key material externally → import into KMS; you manage origin; can set expiry; cannot auto-rotate
Asymmetric keysRSA, ECC for signing/verification or encryption; public key downloadable; private key never leaves KMS
ACM Private CA
  • Fully managed private CA; issue private SSL/TLS certs for internal services
  • Integrate with ACM for EC2, containers, IoT devices
  • Short-lived certificates for zero-trust architectures
  • Audit via CloudTrail; revoke via CRL (Certificate Revocation List) or OCSP
EBS & S3 Encryption
EBS default encryptionEnable per-region account default; new volumes encrypted automatically with specified CMK
EBS snapshot cross-accountShare encrypted snapshot: share CMK + snapshot with target account; target must have CMK decrypt permission
S3 Bucket KeysReduce KMS API calls (and costs) by generating bucket-level DEK; reduces calls by up to 99%
For cross-account encrypted EBS snapshot sharing: both the snapshot AND the CMK must be shared. Target account must be granted kms:Decrypt + kms:ReEncrypt* on the CMK.
Custom-Key-Store ACM-Private-CA EBS-Encryption Key-Import
