D1 · Security & Risk Management

15%
Risk Formulas (Quantitative)
ALE = SLE × ARO
Annual Loss Expectancy = Single Loss Expectancy × Annual Rate of Occurrence
SLE = AV × EF
Single Loss Expectancy = Asset Value × Exposure Factor
Safeguard Value = ALE(before) − ALE(after) − Cost
Positive value = safeguard is cost-effective
Risk Responses
AcceptAcknowledge & tolerate; document residual risk
TransferShift to third party (insurance, contracts)
AvoidEliminate the activity causing the risk
MitigateReduce likelihood or impact with controls
Risk Assessment Methods

QualitativeRisk matrix: likelihood × impact ratings
Semi-quantOrdinal scales (High/Med/Low + numeric)
Policy Hierarchy

Key Concepts

CIA Triad & Security Concepts

Confidentiality
Only authorized parties access data. Controls: encryption, access controls, classification.
Integrity
Data accuracy & completeness. Controls: hashing, digital signatures, version control.
Availability
Systems accessible when needed. Controls: redundancy, RAID, backups, DDoS protection.
Additional Properties
AuthenticityData source is genuine and verified
Non-repudiationCannot deny performing an action (digital signatures)
AccountabilityActions traceable to individuals (audit logs)

D3 · Security Models

Architecture & Engineering — 13%
Formal Security Models
Bell-LaPadulaConfidentiality: no read up (simple), no write down (*-property)
BibaIntegrity: no read down, no write up (opposite of BLP)
Clark-WilsonIntegrity via well-formed transactions & separation of duties
Brewer-NashChinese Wall: prevents conflict of interest across clients
Graham-Denning8 protection rights: create/delete objects & subjects, read/grant/delete/transfer rights
BLP protects confidentiality (military). Biba protects integrity (financial). Clark-Wilson used in commercial environments.
Trusted Computing
TCBTrusted Computing Base: all hardware/software/firmware protecting the system
Reference MonitorAbstract machine mediating all access; must be complete, isolated, verifiable
Security KernelHardware/software implementing reference monitor concept
Security PerimeterBoundary separating TCB from rest of system
Open SystemStandards-based; interoperable
Closed SystemProprietary; vendor-specific

D3 · Cryptography — Keys & Algorithms

Architecture & Engineering — 13%
Symmetric Encryption (same key)
DES56-bit key, 64-bit block — broken, do not use
3DES112/168-bit effective — being phased out
AES128/192/256-bit — current standard (Rijndael)
Blowfish/Twofish64/128-bit block, variable key — fast
RC4Stream cipher — deprecated (WEP/early TLS)
ChaCha20Stream cipher — used in TLS 1.3
Asymmetric Encryption (key pair)
RSASecurity based on factoring large primes; 2048+ bits
ECCElliptic Curve — smaller keys, same strength (256-bit ≈ RSA 3072)
DH/ECDHKey exchange only (not encryption); forward secrecy
El-GamalBased on discrete logarithm; doubles ciphertext size
DSA/ECDSADigital signatures only
Hashing Algorithms
MD5128-bit output — cryptographically broken, collisions found
SHA-1160-bit — deprecated; collision demonstrated 2017
SHA-256SHA-2 family; 256-bit — widely used standard
SHA-512SHA-2 family; 512-bit — highest SHA-2 output
SHA-3Keccak sponge construction; NIST standard 2015
HMACHash + shared secret key for message authentication
Hashing is one-way — cannot decrypt. Used for integrity, not confidentiality.

D3 · PKI & Digital Signatures

PKI Hierarchy
Root CASelf-signed; offline; trust anchor
Intermediate CAOnline; issues end-entity certs; limits Root exposure
End EntityServer/user/device certificates
CRLCertificate Revocation List; periodic; CA-published
OCSPOnline real-time revocation check
Cert PinningApp embeds expected cert/key; bypasses CA chain
Digital Signatures
SignHash the message → encrypt hash with sender's PRIVATE key
VerifyDecrypt with sender's PUBLIC key → compare to hash
Signatures provide authentication, integrity, and non-repudiation. They do NOT encrypt the message content.

D7 · Security Operations

13%
Incident Response Lifecycle (NIST SP 800-61)
1. PreparationPolicies, tools, training, IR plan, playbooks
2. DetectionIdentify indicators; SIEM alerts, IDS/IPS
3. AnalysisTriage, scope, determine nature of incident
4. ContainmentShort-term (isolate) + long-term (patch/rebuild)
5. EradicationRemove malware, close attack vector
6. RecoveryRestore systems; monitor for re-infection
7. Lessons LearnedPost-incident review; update procedures
Digital Forensics
Order of VolatilityCPU regs/cache → RAM → swap/pagefile → disk → remote logs
Chain of CustodyDocument every person who handles evidence
Write BlockersPrevent any modification of evidence drive
Hash EvidenceMD5/SHA hash before & after to prove no tampering
Live ForensicsCapture volatile data before powering off system
SIEM & Log Management
SIEMSecurity Info & Event Management: aggregation + correlation
SOARSecurity Orchestration, Automation & Response
Log RetentionTypically 90 days online, 1 year archived (varies by regulation)
NTP SyncTime synchronization critical for log correlation
BCP / DRP Key Metrics
RTO — Recovery Time Objective
Max acceptable downtime before recovery must complete
RPO — Recovery Point Objective
Max acceptable data loss measured in time
MTD/MTO — Max Tolerable Downtime
Longest acceptable outage before mission failure (RTO ≤ MTD)
MTBF = Total Uptime ÷ # Failures
Mean Time Between Failures — reliability metric
MTTR = Total Downtime ÷ # Repairs
Mean Time to Repair — maintainability metric
Recovery Site Types
Hot SiteFully operational; real-time replication; hours to activate
Warm SiteHardware ready; data partially current; days to activate
Cold SiteFacility only; no equipment; weeks to activate; cheapest
Mobile SiteTrailer/vehicle; rapidly deployed anywhere
DR Testing Methods (least → most disruptive)
TabletopDiscussion only; walk through scenarios verbally
WalkthroughStep through plan; no actual failover
SimulationPractice response without switching to DR site
Parallel TestActivate DR site; primary still operational
Full InterruptionShut down primary; full failover; highest risk/realism
BIA — Business Impact Analysis
PurposeIdentifies critical processes and quantifies impact of disruption
OutputMTD, RTO, RPO values per critical process
BIA is performed BEFORE developing the BCP. It drives all recovery decisions. Senior management owns BCP.
Change Management Process
Request for Change (RFC) Impact Assessment CAB Review & Approval Implementation Testing/Validation Closure & Documentation Emergency Change Process

Common Exam Traps

Risk = Threat × VulnAsset value multiplied in ALE calculation
BLP vs BibaBLP = confidentiality; Biba = integrity
Public key usageEncrypt TO someone; verify their signature
Private key usageDecrypt; CREATE digital signature
RADIUS vs TACACS+TACACS+ = TCP + full encryption; network devices
OAuth vs OIDCOAuth = authorization; OIDC = authentication
RTO vs RPORTO = downtime tolerance; RPO = data loss tolerance
Due Care vs DiligenceCare = doing it; Diligence = verifying it's done
CISSP favors comprehensive, process-oriented answers over quick technical fixes.
