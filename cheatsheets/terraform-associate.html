HCL Syntax — Core Blocks

Block Types
terraform block
terraform { required_version = ">= 1.5" } required_providers { aws = { source = "hashicorp/aws" version = "~> 5.0" } }
provider block
provider "aws" { region = "us-east-1" alias = "east" }
resource block
resource "aws_instance" "web" { ami = var.ami_id instance_type = "t3.micro" }
data source block
data "aws_ami" "latest" { most_recent = true owners = ["amazon"] }
Block Reference
terraform {}required_version, required_providers, backend
provider {}source, version, alias, config args
resource {}type, name, arguments, meta-args
data {}source type, name, filters → read-only
variable {}type, default, description, validation, sensitive
output {}value, description, sensitive, depends_on
locals {}computed values, avoid repetition
module {}source, version, input vars
Reference syntax: resource → aws_instance.web.id | data → data.aws_ami.latest.id | var → var.ami_id | local → local.name | module → module.vpc.subnet_id

Backends · Locking · Commands
State File
Default locationterraform.tfstate (local JSON)
PurposeMaps real resources to config
Sensitive dataStored in plaintext — secure your backend!
terraform.tfstate.backupPrevious state auto-saved
Remote Backends
S3 + DynamoDBState in S3, locking via DynamoDB
Terraform CloudRemote runs + state + teams

State Commands
terraform state listList all resources in state
terraform state show ADDRShow resource attributes
terraform state mv SRC DSTRename/move resource in state
terraform state rm ADDRRemove resource from state (no destroy)

terraform state pushUpload state to backend

State locking prevents concurrent terraform apply runs. Always use DynamoDB (AWS), leases (Azure/GCS), or TFC for team environments.

Workspaces

Isolation · Commands · Best Use
Workspace Commands
terraform workspace new staging terraform workspace list terraform workspace select staging terraform workspace show terraform workspace delete staging
Using Workspace in Config
resource "aws_s3_bucket" "app" { bucket = "myapp-${terraform.workspace}" }
Key Facts
Default workspaceAlways exists, cannot delete
State isolationEach workspace has separate state file
NOT for prod/stagingUse separate backends/configs instead
Best forFeature branches, dev variants, testing
terraform.workspaceInterpolation variable with current name
Workspaces share the same backend config. For true environment isolation (prod vs staging), use separate state backends or different Terraform directories.

Provider Locking & Versioning

Lock File · Constraints · Aliases
Lock File
.terraform.lock.hclAuto-generated, commit to Git
terraform providers lockPre-populate lock for multiple platforms
PurposeEnsures team uses same provider versions
-upgrade flagUpdates lock file to newer allowed versions
Version Constraint Operators
= 1.0.0Exact version only
!= 1.0.0Exclude this version
> 1.0, < 2.0Range constraint
>= 1.0.0This version or higher
~> 1.0.0Pessimistic: >= 1.0.0, < 1.1.0
~> 1.0Pessimistic: >= 1.0, < 2.0
Provider Configuration
terraform { required_providers { aws = { source = "registry.terraform.io/hashicorp/aws" version = "~> 5.0" } } }
Provider sourceregistry.terraform.io/namespace/type
Default registryregistry.terraform.io (can omit prefix)
Provider Aliases
provider "aws" { alias = "west" region = "us-west-2" } provider = aws.west
Use caseMultiple regions or accounts
In modulePass via configuration_aliases in required_providers
Always commit .terraform.lock.hcl. Run terraform providers lock -platform=linux_amd64 -platform=darwin_amd64 in CI to ensure cross-platform consistency.

Quick Reference — Key Concepts

Exam Tips & Gotchas
Variable Precedence (highest → lowest)
1. -var flagterraform apply -var="name=val"
2. -var-file flagterraform apply -var-file=prod.tfvars
3. *.auto.tfvarsAuto-loaded alphabetically
4. terraform.tfvarsAuto-loaded if present
5. TF_VAR_nameEnvironment variable
6. default valueDefined in variable block
Terraform Refresh & Plan Symbols
+ (green)Resource will be created
- (red)Resource will be destroyed
~ (yellow)Resource will be updated in-place
-/+ (red/green)Resource will be destroyed and re-created
<= (cyan)Data source will be read
Important File Names
main.tfPrimary configuration (convention)
variables.tfInput variable declarations
outputs.tfOutput value declarations
versions.tfProvider version constraints
terraform.tfvarsVariable values (auto-loaded)
.terraform/Plugin cache dir — do NOT commit
.terraform.lock.hclLock file — COMMIT to Git
terraform.tfstateState file — do NOT commit (use remote)
Common Exam Gotchas
terraform.tfstateNever commit to Git — use remote backend
count vs for_eachfor_each preferred — handles deletions better
terraform taintDeprecated — use terraform apply -replace
terraform refreshDeprecated standalone — use plan -refresh-only
Backend changesRequire terraform init -reconfigure
No "terraform push"Use TFC VCS integration instead
The terraform.tfvars file and any *.auto.tfvars files are automatically loaded. Files with other names must be explicitly passed with -var-file. TF_VAR_ env vars are the lowest explicit override.
